## C++ 开发总原则

- 使用 C++17 标准。
- 优先考虑运行效率与工程可维护性。
- 避免不必要的对象拷贝，优先使用引用、移动语义和合适的返回值优化。
- 资源管理优先使用 RAII。
- 当所有权明确且需要动态生命周期管理时，优先使用智能指针，例如 `std::unique_ptr`。
- 在不破坏现有架构的前提下，尽量保持接口清晰、职责单一、依赖方向稳定。

## 面向数据设计 (DoD) 与内存布局规则

- 核心计算流程坚持面向数据设计 (Data-Oriented Design)，避免传统“大对象包裹数据”的重度封装。
- 所有的物理场（如位移、温度、力等）及状态变量必须经由 `FieldManager` 统一管理，并以连续的 SoA (Structure of Arrays) 布局（即连续的 `double*` 一级指针，长度为 `NUM_PARTICLES`）在内存中开辟。
- 严禁在粒子遍历、受力计算、邻域计算等高频/密集计算的核心循环中引入 AoS (Array of Structures) 型的重度粒子对象，以消灭指针追逐 (Pointer Chasing)，榨干 CPU L1/L2 缓存。
- 在高性能计算 Kernel 内部，应优先获取连续场数据的裸指针（Raw Pointer）并直接对其进行指针偏移与数学计算，确保与 OpenMP 并发高效协同。

## 性能与并行

- 对计算密集型循环，例如粒子遍历、受力计算、邻域相关计算等，优先评估是否适合使用 OpenMP 并行化。
- 当循环之间不存在数据竞争，且并行收益明显时，优先考虑 `#pragma omp parallel for`。
- 引入或修改并行逻辑时，必须同时考虑：
  - 线程安全
  - 数据竞争
  - 共享变量与私有变量划分
  - 累加、归约、缓存局部性
  - 并行开销是否超过收益

- 不要为了“使用并行”而并行；如果循环规模很小、依赖复杂或串行更稳妥，则保持串行实现。

## CMake 与构建约束

- 保持项目使用 C++17，并确保相关 CMake 配置一致。
- 如果任务涉及 OpenMP，必须同时检查并维护 CMake 中的 OpenMP 配置，确保构建链路完整。
- 不随意引入新的第三方依赖，除非任务明确要求且理由充分。
- 修改构建脚本时，优先保持与现有工程结构兼容，不做无关重构。

## 模块设计模式

- 对核心模块，例如 `Field`、`Material`、`BC`，统一遵循以下模式：
  - `Manager`：统一持有并管理该模块的实例。
  - `Registry`：提供全局注册中心，并支持编译期静态注册。
  - `Factory`：通过字符串类型名动态创建具体子类对象。
  - `Singleton`：注册表通常实现为单例，例如 `FieldRegistry::getInstance()`。

- 新增同类模块能力时，优先沿用既有模式，不随意引入风格不一致的新机制。
- 如果现有实现已经形成稳定约定，优先贴合现有代码风格，而不是抽象出“理论上更通用”的新架构。

## 模块解耦原则

- 模块之间必须保持高度解耦。
- 材料本构模型不应直接依赖具体求解器核心。
- 各模块之间优先通过抽象基类、稳定接口和多态进行协作。
- 避免跨模块直接访问内部细节，避免形成反向依赖、循环依赖或隐式耦合。
- 当需要扩展功能时，优先在接口层扩展，而不是在实现层硬编码耦合逻辑。

## 8层计算管线调用与隔离原则

- 整个引擎计算流转严格遵循 8 层执行体系，防止面条式代码 (Spaghetti Code) 产生：
  1. **层 1：输入解析控制层** (`IOManager`, `MeshReader` 等解析 YAML 与网格数据)
  2. **层 2：预处理与内存初始化池** (`ParticleManager`, `NeighborList` 等建立拓扑连接)
  3. **层 3：大容量连续场接管层** (`FieldManager` 动态开辟 SoA 物理场)
  4. **层 4：时间积分与外力驱动器** (`TimeIntegrator` 执行时间步推进与施加 `BCManager` 边界)
  5. **层 5：多体接触与摩擦防卫层** (`ContactManager` 运行防穿透与库仑摩擦)
  6. **层 6：物理积分核方程体系** (`PDKernel` 处理非常规态基 NOSB-PD 张量求逆及稳定器惩罚)
  7. **层 7：力学和物理本构层** (`Material` 本构计算应力/热通量响应值)
  8. **层 8：损伤演化与拓扑断裂层** (`DamageModel` 评估断键阈值并修改近邻拓扑)
- 每一层应严格执行单一职责，同层级内部通过抽象基类多态隔离，上下层之间通过标准接口或统一数据总线（如 `PDContext`）协作，严禁越级或逆向强耦合依赖（例如，底层材料本构 `Material` 严禁反向依赖求解器 `PDEngine` 或积分器 `TimeIntegrator`）。

## 物理场生成规则

- 通过物理问题类型生成对应的待求解物理场，例如 `Thermal`、`Mechanical`。
- 使用 `PhysicsFieldRegistry` 与 `PhysicsFields` 子类，自动向 `FieldManager` 注册所需字段。
- 新增物理问题类型时，优先沿用“Physics Type -> Fields”的既有注册与生成路径。
- 不要绕过注册体系手工散落地创建字段，除非任务明确要求且现有架构无法承载。

## 材料状态变量规则

- 通过材料类型生成对应的内部状态变量场。
- 每个 `Material` 子类应实现 `allocateStateVariables(FieldManager&)`。
- 新增材料类型时，应同步考虑其状态变量声明、分配、注册与使用链路是否完整。
- 不要把材料内部状态变量的生成逻辑散落到求解器或其他无关模块中。

## 准静态与隐式求解环路安全规则

- 在编写或优化 `TimeIntegrator`（如自适应动态弛豫 `ADR_Integrator` 或无矩阵隐式 `MatrixFreeImplicitIntegrator`）的求解逻辑时，必须严格区分 **试探内循环 (Inner Loop)** 与 **外层载荷步循环 (Outer Loop)**：
  - 在试探内循环期间，材料本构必须处理好“状态冻结/解冻 (State Frozen/Unfrozen)”机制（例如，设置 `stateMode` 并在只读/只算弹性试探态下工作），不得在未收敛时修改或污染历史状态变量。
  - 只有在外层确定收敛（即宏观力/位移残差达标）或载荷步结束时，才可触发 `commitState()` 进行物理历史数据真实落盘，并调用 `DamageModel` 进行正式的损伤断裂判定与拓扑断键。
- 隐式计算中对极易发生 `NaN` 崩溃的非线性迭代（如 Lemaitre 隐式损伤积分），优先采用解析解（或高度稳定的解析截断）替换传统的牛顿-拉夫逊 (Newton-Raphson) 迭代，确保深层断裂时的绝对数值稳定性。

## 接触、摩擦与几何降维量纲规则

- 接触子系统设计必须遵循“几何搜索与算法（Type，如 NTN/NTS）”与“力学交互公式（ForceLaw，如 Penalty/Silling）”双轴正交注册模式，确保高阶模型与排斥力学算子能够积木式灵活组合。
- 在涉及 2D 模拟（Plane Stress / Plane Strain）时，接触刚度与近场动力学微模量计算必须引入厚度修正或维度降维因子（如二维微模量公式修正：$c_{2D} = \frac{9K}{\pi t \delta^3}$），避免量纲失真。
- 摩擦力（如 Coulomb 摩擦力）计算必须施加单步冲量限幅锁死机制（如 $F_{max} = \frac{m^* v_t}{dt}$），以在数值积分层面上彻底根除静、动摩擦转态（Stick-Slip）时的反向自激抖动和爆炸式跳弹。

## 抗沙漏与零阶能震荡控制规则

- 针对非常规态基近场动力学 (NOSB-PD)，由于其微观积分性质，极易在点阵离散空间内诱发零阶伪能（Hourglass / Zero-Energy）震荡。
- 代码中必须配备并正确注册专用的稳定器（如 `Zhang`、`Wan` 或 `Silling` 稳定器），在力学积分核（`NOSB_M`）计算中通过张量投影阻力阵列或刚度缩并实施有效的人工稳定力补偿。
- 新增或调整计算核物理方程时，必须确保零能稳定控制逻辑的激活，不得裸算无稳定控制的态基模型。
